26 #### Positioning and Sizes
27 
28 Unlike conventional UI libraries, Consor follows a more 'automated' way of positioning controls.
29 Controls (those that derive from `Consor::Control`) have no position information relating to that control; everything is positioned using containers. Containers well define how and/or where a control will be drawn, and although it may not determine the controls size, it can influence it. For example, a label's size can't be smaller than the message it holds, however, if you `ForceResize(const Size&)` the control, the label will attempt to obey the width, but not the height. EG:
30 
31 ```c++
32 Label label;
33 label.SetText("Hello, world! How are you on this fabulous day?");
34 
35 cout << "label.GetSize() = " << label.GetSize() << "\n";
36 ```
37 
38 The result of the operation above should yield the output:
39 
40 ```
41 label.GetSize() = 48, 1
42 ```
43 
44 Now, let's say that the label's width should not exceed a width of 30 characters, we can `ForceResize(const Size&)` the label, and it will wrap its text at the most appropriate places:
45 
46 ```c++
47 Label label;
48 label.SetText("Hello, world! How are you on this fabulous day?");
49 label.ForceResize(Size(30, 1));
50 
51 cout << "label.GetSize() = " << label.GetSize() << "\n";
52 ```
53 
54 The program above should yield the output:
55 
56 ```
57 label.GetSize() = 28, 3
58 ```
59 
60 As we can see, the control's size is now acceptable, but the control is not the desired size (`30, 1`); this is because a control does not need to obey your force resize instructions, and it is up to you to verify that the control is of a suitable size.
61 
62 Another, important rule you need to know, is that a control may not render outside of it's designated area. The drawing of elements is strictly positioned relativly, and any attempt to draw outside of these bounds will be clipped and nullified. The controls position will be decided over a cascade of containers (and possibly other controls), of which will arive for you to draw at 0, 0 with the width and height of your control.
63 For example, a `ScrollContainer` whose client is a `FlowContainer` will result in your controls position being none-static, but you controll SHOULD never see this level of abstraction.
64 
65 Please note, an excepion will NOT be thrown upon rendering outside of the render bounds; this is intended behavour, primarily for use with the `ScrollContainer`.
66 If you absoloutly must, and your control does not influence other controls (such as a tool tip), you may manually adjust the render bounds stack, but this is not advised and strongly discouraged as compatability may break, or it could interfere with other controls.

97 #### Input System
98 
99 The input system provides the standard input Consor will need, and must implement these pure methods:
100 
101 ```c++
102 using namespace Consor::Input;
103 
104 bool KeyWaiting(); // is a key waiting on the input buffer?
105 Key GetKeyPress(); // get the next key press (repeats if held)
106 bool ControlDown(); // is control currently pressed?
107 bool ShiftDown(); // is shift currently pressed?
108 ```
109 
110 Upon input, an event will be generated at the highest level. The event will be passed (recursively) to all clients (down `Control::HandleInput(Key, IInputSystem&)`); if all clients do not handle the input, then the control checks if itself should handle the input.
111 A control may steal the input before passing to their clients, such as scrolling a `ScrollContainer` when control is pressed, rather than focus being shifted.
112 
113 An example of how this would work with a `TextBox`, and a `ScrollContainer` is shown below:
114 
115 ```
116 Structure of top:
117 top
118  ScrollContainer
119  FlowContainer
120  TextBox
121  Button
122 
123 Key = 'a'
124 top
125  -> ScrollContainer
126  -> FlowContainer (Currently focused: TextBox)
127  -> TextBox
128  <INPUT HANDLED - Key typed>
129 
130 Key = Key::Down
131 top
132  -> ScrollContainer
133  -> FlowContainer (Currently focused: TextBox)
134  -> TextBox
135  <UNHANDLED>
136  <-
137  <INPUT HANDLED - Focus changed to Button>
138 
139 Key = Key::Enter
140 top

164 #### The Window System
165 
166 The current Window System that is inbuilt into Consor positions any control into a desirable, or requested position.
167 
168 A position requested of -1, -1 should be centered, and those in the range of 0 to 1 (0 to 1 not implimented yet) should be positioned said percentage along the screen. So `(-1.0, 2.0/3.0)` will be positioned center X, and 2/3rds down the screen.
169 
170 The idea of `WindowSystem` is to provide a simple and generic way to create a window for the user (for Prompts, etc...), you may expand or replace `Consor::WindowSystem` with your own.
171 The easiest way to achive this, would be to create a blank control of the same size of the console (`IConsoleRenderer::GetSize()`) and use it as your base window system. This keeps compatibility with prompts (`MessageBox`, `ChoiceList`, and `InputBox`) and any other applications that may use the same instance of `WindowSystem`, while allowing you to build upon your own canvas.
172 
173 The window system also impliments support for hot keys via the use of `WindowSystem::RegisterHotKey(Control* pControl, Input::Key Key, bool Shift, bool Ctrl, std::function<void()> Callback)`. Upon the key combo being pressed, `Callback` will be invoked; if `WindowSystem::UnregisterWindow(Control* pControl)`'s pControl matches an existing hot key, then the hot key will be removed.
